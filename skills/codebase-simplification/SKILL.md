# Codebase Simplification

継続的なコードベース整理の判断基準を提供する。
監査手順や特定アーキテクチャを正解として固定せず、候補の採否・方向・作業単位を同じ語彙で決める。

分類全体を説明するときは [decision-map.png](assets/decision-map.png) を概要図として使い、repository固有の証拠と混同しない。

## 北極星

業務仕様と設計判断の正本を基準に、正本から説明できない仕様、概念、状態、分岐、依存、遠回りを削除または再配置する。

成果は、1つの変更のために人とAIが読む・変える・壊しうる範囲、すなわち `working set` の純減で測る。
LOCは結果指標の1つとして扱い、圧縮や巨大関数化を削減と数えない。

## 最初に正本と制約を確定する

1. repository rule、業務仕様、ADR、DB・API・認可契約、進行中のIssue / PRを読む。
2. 対象を hard constraint、soft convention、accidental structure に分類する。
3. 正本が不足する場合は、削除や意味変更を確定せず、必要な判断をユーザーへ返す。
4. 既存Issue / PRとの重複、依存関係、競合する作業範囲を確認する。

## 上位から疑う

次の順で整理対象を調べる。
上位で消せるものを、下位で磨かない。

1. 仕様・業務能力
2. データ・状態・永続化
3. ルール・分岐・error・fallback
4. 構造・抽象化・中間層・境界・依存
5. surfaceとなるUI・API・test・docs・設定

仕様ごと不要ならsurfaceを整えず削除する。
状態を持つ必要がなければ、その状態を扱う分岐やhelperを先に直さない。

## 指摘から入るときも上位から疑う

ユーザーやレビューが症状(重複、長い関数、似たコード)を指して始まる点検でも、同じ順序で疑う。

1. 指摘の語彙が示す操作をそのまま採用しない。「重複している」を統合と直結させず、候補を `削除 → 畳む → 統合…` の順へ通す。
2. 症状を磨く前に、その症状を生んだ上流の形を1階層以上遡って特定する。guard、lookup、ID結合、詰め替え、並行構造の突き合わせ、防御的throwは、上流のデータ形・出力形が生んだ症状であることが多い。上流の形を変えるとその操作ごと消えるなら、それを第一候補にする。
3. 最初に成立した操作で確定しない。より上位で消す案を先に検討し、不成立の理由を言えてから下位の操作を採用する。

歯止めも同時に守る。

- 上位遡りは、replacement込みでworking setと概念が純減する場合だけ採用する。
- 上流に生成元のない、形だけの重複(style定義、設定値など)は、共通部分の統合で1正本化して止まる。
- 統合を第一候補にする場合、公開面とmarkupを保つ最小の1正本化にする。参照方式の変更や構造の付け替えは、独立した候補へ分ける。

## 4つの品質軸で望ましい状態を定義する

### 1. 意味のSingle Source of Truth

同じ業務上の意味を複数箇所で定義しない。
同じ形のコードではなく、同じ理由で変わる意味を統合する。
層ごとに表現が異なる場合は、正本と変換地点を明示する。

### 2. 変更局所性

1つの業務挙動を理解または変更するとき、必要な入力・状態・ルール・error・testを、その正本を持つmoduleと直接依存の中で完結させる。

次を減らす。

- 読むfile数
- 呼び出しhop数
- またぐlayer数
- 同じ変更理由で触るmodule数

凝集度はmodule単体、結合度は依存の強さを測る。
変更局所性は「1つの変更タスク」を単位に、変更経路の長さとworking setを測る。

### 3. 最小capability

consumerへwide DTO、巨大context、repository全体、不要なcallbackを渡さない。
必要な値または能力だけを渡し、未使用入力とバケツリレーを削除する。

### 4. 不正状態の拒否

正しい判別子、型、DB制約、入口のvalidationで不正状態を早く拒否する。
leafごとのguardやfallbackで不正状態を延命しない。
状態の組み合わせを型で表せる場合は、実行時guardを増やす前に型と永続化境界を検討する。

不正状態を拒否する仕組みの追加だけでworking setが減らない場合は、整理候補ではなくbug、feature、guardrail改善として分流する。
入口の制約によってleaf guard、fallback、不要状態を消せる場合は整理候補に含める。

## 6つの操作へ分解する

候補を次の動詞のいずれか1つで表す。
1語で表せない候補は分解する。
どの動詞にも収まらず、消えるものもない候補は別種の改善へ分流する。

- **削除する**: 不要仕様、状態、fallback、互換経路、死んだsurfaceを消す。
- **畳む**: 垂直方向のpass-through wrapper、詰め替え、不要hopを呼び出し経路から消す。
- **統合する**: 水平方向に重複する同一概念、正本、実装を1つへまとめる。
- **移動する**: ルール、error handling、状態の所有権を正しいmoduleまたは入口へ寄せる。
- **狭める**: 型、入力、公開面、依存、権限、変更可能範囲を必要最小限にする。
- **分離する**: 異なる変更理由、ライフサイクル、ドメインを切り離す。

原則として `削除 → 畳む → 統合・移動・狭める・分離` の順で検討する。
価値を確認していないコードへ新しい住所や抽象を与えない。

## 候補を判定する

詳細な質問と用語は [judgment-framework.md](references/judgment-framework.md) を読む。

候補ごとに次を満たす。

1. 正本または業務価値との関係を説明する。
2. 操作を6動詞の1つで表す。
3. 消える概念・状態・分岐・依存・hop・LOCを数える。
4. 変更前後の観察可能な挙動を明示する。
5. 判断を business、design、mechanical のいずれかに分類する。
6. hard constraint、migration、可逆性、データ影響を確認する。
7. 再発防止guardと、別経路へ逃げる余地を確認する。
8. やらない場合の具体的なfailure modeまたは将来コストを示す。
9. 1つの読み筋としてレビュー可能な単位へ切る。

価値の不在を理由に削除する場合は、実装参照、UI / API到達性、業務正本など3つの独立ソースで裏を取る。
grep件数だけで不要と判定しない。

## 挙動変更を独立したリスクとして扱う

「既存挙動を変えない」を一律の制約にしない。
不要挙動の削除や正しい仕様への収束も整理に含める。

ただし、候補ごとに次を明示する。

- 変わる挙動
- 維持する契約
- 影響を受ける利用者とデータ
- migrationまたは段階移行の要否
- 判断者と承認条件
- 検証方法とrollback方法

業務価値を削る判断、不可逆なデータ変更、公開契約の破壊は、ユーザーの確認前に実装しない。

## 抽象化と共通化を目的にしない

共通化は、意味の正本と変更理由が同じ場合だけ採用する。
偶然似たコードは局所に残す。

moduleの削除を想像する `deletion test` を使う。

- 削除すると複雑さ自体が消えるなら、pass-throughの可能性が高い。
- 削除すると同じ複雑さが複数consumerへ再出現するなら、moduleは意味を隠している可能性が高い。

interfaceの実装が1つという事実だけで削除しない。
依存方向、所有権、test seam、外部systemとの差し替えという価値を確認する。
1つのadapterしか想定できないseamは投機的である可能性を疑い、2つ以上の実在adapterがあるseamを優先する。

## 配置ポリシーを先に確定する

統合・移動を含む候補が複数見込まれる場合、候補ごとに置き場所を決めない。
先に配置ポリシーを確定して記録する。

配置ポリシーは設計文書だけから導かない。
文書は実態より古いことがある。
実在するディレクトリと模範実装を読み、次をセットで決める。

- 層ごとの置き場所と、基準例となる実在module
- 共通部品へ昇格する条件
- 依存方向と禁止import
- common / utilsなど、所有する意味を示さない名前の禁止

配置ポリシーを確定または変更したときは、確定前に判定した候補と作業単位を新しいポリシーで再判定する。

## 作業単位を設計する

- 1 PRを1つの読み筋にする。
- 変更する契約と維持する契約を分ける。
- review順を明示する。
- 非競合なら並列、依存するならstacked PRにする。
- current HEADの検証を使い、人間レビューなどrepository固有のmerge条件を守る。
- 行数目標のために必要なtest、guard、説明を削らない。

## ゴールを定義する

長期の整理活動では、次の形を基礎にrepository固有の正本、merge条件、停止条件を加える。

```text
<repository>で、正本の仕様と業務価値を基準に、不要な仕様・挙動・データ構造・状態・分岐・中間層・fallback・互換経路・依存を大胆に削減する。
同じ意味は1つの正本へ集め、consumerには必要なcapabilityだけを渡し、不正状態は型・DB・入口で拒否する。
1変更あたりに読むfile、呼び出しhop、またぐlayer、変更対象moduleを減らして変更局所性を高め、概念・分岐・依存・net LOCの純減を評価する。

候補は正本、ADR、既存Issue / PR、利用実態と照合し、変わる挙動、維持する契約、migration、可逆性、同梱guard、検証方法を明示する。
1 PR 1読み筋へ分割し、非競合なら並列、依存するならstackedにする。
<repository固有のreview / CI / merge条件>を守り、証拠を伴う実行価値のある候補がなくなるまで継続する。
```

候補の出力には次を含める。

```text
対象 / 正本:
操作:
品質軸:
現状のworking set:
消えるもの:
変わる挙動 / 維持する契約:
制約・migration・可逆性:
証拠:
同梱guard:
PRの読み筋と依存:
採用 / 見送り / 要判断:
```

## 停止と見送りを記録する

候補を proposal、tracked、rejected、insufficient-evidence のいずれかへ着地させる。
見送り理由を残し、同じ候補を毎回再導出しない。

次のいずれかで停止する。

- 証拠を伴い、実行価値が閾値を超える新規候補が出なくなった。
- 残る候補が正本不足またはbusiness decision待ちになった。
- 競合中の作業へ統合する方が合理的になった。
- ユーザーが打ち切りまたは優先度変更を指示した。

## 他skillと分担する

利用可能な場合に限り、次を組み合わせる。

- `bold-maintainability-audit`: repository全体から候補と定量証拠を発見する。
- `constraint-boundary-review`: hard / soft / accidental constraintを精査する。
- `domain-modeling-interview`: 統合・分離・仕様削除の前に意味の正本を確定する。
- `minimum-diff-convergence`: 採用後の1 PRを最小のreview単位へ収束させる。
- `shida-review-report`: 差分をレビューし、findingと図解を成果物へ残す。

具体例と負例が必要な場合は [examples.md](references/examples.md) を読む。
